Helm Chart 发布到 Harbor OCI 仓库

Harbor 可以使用 OCI Distribution 协议保存 Helm Chart。Chart 与容器镜像共用 Harbor 的项目、身份认证、权限控制、复制和审计能力，不需要维护单独的 Helm 仓库索引。

Chart 通过 Harbor OCI 完成发布与消费。Chart 目录、模板、Values 和依赖管理参阅《Helm：Kubernetes 包管理与 Chart 开发》。

一、OCI 与传统 Helm Repository

传统 Helm Repository 由 HTTP 文件服务提供 .tgz 包和 index.yaml：

OCI 仓库按照 artifact 引用直接访问 Chart：

  • 传统仓库依赖 index.yaml 维护名称、版本和下载地址
  • OCI 使用 registry API，通过 repository 与 tag 定位 artifact
  • 传统仓库使用 helm repo add 和 helm repo update
  • OCI 使用 helm registry login，消费时直接传入 oci:// 引用
  • Harbor OCI 可以复用项目权限、Robot Account、审计和复制策略

OCI 改变的是 Chart 的分发方式，不改变 Chart 模板的渲染和 Release 管理机制。

二、示例环境

示例环境：

  • Helm 3.8 或更高版本
  • Harbor 中已经创建项目 helm
  • 发布身份对项目具有推送 artifact 的权限
  • 部署身份具有拉取权限
  • 客户端信任 Harbor 的 HTTPS 证书
  • 已完成 my-app Chart 的开发和验证

检查 Helm 版本：

Helm 3.8 起 OCI 功能已默认启用，不需要设置早期版本使用的实验性环境变量。

三、Harbor HTTPS 证书

Harbor 使用公开 CA 签发的证书时，Helm 通过系统信任库完成验证，不需要 --ca-file。

Harbor 使用自签名证书或内部私有 CA 时，客户端需要信任对应的 CA。CA 已加入操作系统信任库时，同样不需要 --ca-file。

CA 没有加入系统信任库时，可以使用变量保存证书路径：

HARBOR_CA_FILE 不是 Helm 内置变量，只用于保存私有 CA 证书路径。Helm OCI 命令通过 --ca-file "$HARBOR_CA_FILE" 读取该证书。

CA 证书不是密钥，可以通过受控配置分发。CA 的来源和指纹仍需验证。生产环境不使用 --insecure-skip-tls-verify 或明文 HTTP。

回滚：

helm rollback 的数字是 Release revision，不是 Chart 版本。Helm 不会重新选择 Harbor 中的最新 Chart。

helm rollback 使用 Release 历史恢复 Helm 管理的 Kubernetes 资源和 Values。已经执行的数据库迁移和外部系统变更不会跟着回滚。
