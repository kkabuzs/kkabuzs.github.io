Helm：Kubernetes 包管理与 Chart 开发

Helm 是 Kubernetes 的包管理器。Helm 将 Kubernetes 资源、配置参数和版本元数据组织成 Chart。Helm 以 Release 为单位管理安装、升级与回滚。

一、Helm 解决的问题

Kubernetes 资源较少、环境差异很小时，可以直接维护 YAML。资源和环境增多后，重复配置与版本管理会变得困难：

  • Deployment、Service、Ingress 等资源需要同时发布
  • 开发、测试、生产环境维护了多份只有参数不同的清单
  • 应用升级后，集群中的配置版本难以确认
  • 回滚依赖人工保存旧文件
  • 多个团队重复复制资源定义

Helm 将资源结构放入 Template，将环境差异放入 Values。每次安装或升级都会生成 Release revision。Helm 可以查询 Release 历史。Helm 可以回滚到指定 revision。

Helm 不替代 Kubernetes 控制器。Helm 生成并提交资源。Kubernetes 负责调度、健康检查和持续协调。

二、核心概念

2.1 Chart

Chart 是包含 Template、默认 Values、依赖和元数据的软件包。

2.2 Release

Release 是 Chart 在目标集群和命名空间中的安装实例。同一个 Chart 可用不同 Release 名称和 Values 安装多次。升级 Release 会创建新的 revision。helm rollback 恢复历史 revision，不修改 Repository 中的 Chart。

2.3 Values

Values 是 Template 的输入。Chart 的 values.yaml 保存默认值。Values 文件和命令行参数可以覆盖默认值。副本数、镜像、资源配额和功能开关适合放入 Values。固定 API 结构和标签规则不全部参数化。参数过多会增加测试组合。

2.4 Template

Template 是包含 Go Template 表达式的 Kubernetes YAML。Helm 在客户端渲染 Template，再提交 Kubernetes API。Template 组织资源，不承载复杂业务逻辑。

2.5 Repository

Repository 用于存储和分发 Chart。传统 Repository 通过 HTTP 提供 Chart 包和 index.yaml。OCI Repository 使用 OCI 协议保存 Chart。

使用 Harbor OCI 发布 Chart 可参阅《Helm Chart 发布到 Harbor OCI 仓库》。

4.2 upgrade --install 的行为

CI 使用 upgrade --install。Release 不存在时安装，已经存在时升级。

升级：

  • --wait 等待工作负载和资源就绪
  • --atomic 在升级失败时自动回滚，并隐含 --wait
  • --timeout 设置等待上限

--atomic 和 --wait 不替代应用健康检查与发布后验证。

六、Chart.yaml 与 values.yaml

6.1 Chart.yaml

version 是 Chart 包版本。Chart 内容变化时提升该版本。appVersion 是应用版本，主要用于展示。两个字段不会互相更新。

appVersion 使用引号，避免 YAML 将 1.0 等值转换成数字。镜像标签通过 Values 传入。Helm 不会自动使用 appVersion 作为镜像标签。

6.2 values.yaml

默认 Values 保持可运行。值名保持稳定和明确。密码、Token 和私钥不写进默认 Values，使用 Kubernetes Secret 保存。

按锁定版本恢复依赖：

Chart.lock 提交到版本控制，开发环境和 CI 使用相同的依赖版本。离线构建可以将依赖包一并提交。发布流水线从固定来源取得依赖包，便于审计。

Chart.lock 不检查依赖安全。CI 仍需检查来源、许可证和已知漏洞。

十一、打包与版本管理

恢复依赖、检查并打包：

version: 0.1.0 生成：

查看包内元数据和默认 Values：

Chart 版本通常遵循语义化版本：

  • 修复 Template 或默认 Values：提升 patch
  • 增加向后兼容的 Values：提升 minor
  • 引入不兼容的 Values 或资源变更：提升 major

已发布的 Chart 版本保持不可变。Chart 修改后提升 version 并重新发布。覆盖同版本包会破坏审计与回滚的确定性。

十二、版本、权限与回滚

  • 固定 Chart、镜像和依赖版本。浮动标签会影响运行环境。
  • Chart 与环境 Values 分开。
  • 使用 helm lint、helm template 和服务端 dry-run 分层检查。
  • 为工作负载配置 readiness、资源请求和发布超时。
  • 升级使用 --atomic --wait --timeout。发布后执行业务验证。
  • 回滚前检查数据库迁移和持久化数据的兼容性。
  • 敏感值不进入 Chart、Values、命令行和 CI 日志。
  • Chart.lock 纳入版本控制。CI 检查依赖和镜像。
  • 发布身份权限最小化。Helm 命令明确指定 Release、namespace 和 version。
  • Template 保持简单，只参数化真正变化的内容。
  • Chart 保存 Template 和默认 Values。环境 Values 保存部署差异。Release 保存安装历史。
