CUDA 核心库：cuBLAS、cuDNN 与 NCCL

一，介绍

cuBLAS、cuDNN 与 NCCL 都属于NVIDIA GPU 软件生态。

  • cuBLAS：负责矩阵计算
  • cuDNN：负责神经网络算子
  • NCCL ：负责 GPU 间通信

这些库不是取代CUDA，而是建立在 CUDA 能力之上，为常见任务提供高度优化的实现。

二，cuBLAS：矩阵计算库

cuBLAS = CUDA Basic Linear Algebra Subprograms
它是 NVIDIA 提供的 GPU 线性代数库，可以理解为运行在 NVIDIA GPU 上的 BLAS。
BLAS 是一套标准化的基础线性代数接口，传统 CPU 版本包括：

  • Intel MKL
  • OpenBLAS
  • BLIS

而 cuBLAS 是针对 NVIDIA GPU 优化的实现。

  • 推荐系统
  • 图像处理

2.3 cuBLASLt

cuBLASLt 是更加灵活的矩阵乘法接口。

支持：

  • FP32、FP16、BF16、TF32、FP8 等数据类型
  • Tensor Core
  • 矩阵布局设置
  • Bias 融合
  • 激活函数融合
  • 算法选择与启发式搜索
  • Workspace 管理

2.4 cuBLAS 的注意事项

小矩阵不一定适合 GPU
如果矩阵非常小：

  • Kernel 启动有开销
  • CPU 与 GPU 数据传输有开销
  • GPU 并行资源无法充分利用

因此 GPU 不一定比 CPU 快。

Tensor Core 的使用
要高效使用 Tensor Core，通常需要：

  • 合适的数据类型
  • 合适的矩阵尺寸和对齐方式
  • 正确选择计算模式
  • 足够大的工作负载

三，cuDNN

3.1 cuDNN 是什么

cuDNN = CUDA Deep Neural Network library
它是 NVIDIA 提供的深度神经网络 GPU 加速库。
cuDNN 提供的不是一个完整的神经网络框架，而是经过高度优化的底层算子实现。

3.2 cuDNN 支持的典型算子

包括但不限于：

  • 卷积
  • Pooling
  • 激活函数
  • Softmax
  • Batch Normalization
  • 张量变换
  • Attention
  • RNN
  • 融合算子

3.3 为什么需要 cuDNN

以卷积为例，同一个数学操作可以有多种实现方法：

  • Direct Convolution
  • Implicit GEMM
  • Winograd
  • FFT
  • 针对特定形状优化的 Kernel

不同算法适合不同情况：

  • 输入张量大小不同
  • 卷积核大小不同
  • 数据类型不同
  • GPU 架构不同
  • 可用 Workspace 不同
  • 对确定性和精度的要求不同

cuDNN 会根据具体条件选择合适的执行方案。

因此，cuDNN 的价值不仅是“实现了卷积”，更重要的是：

为不同 GPU 和张量形状提供经过高度优化的算法与执行方案。

四，NCCL

4.1 NCCL 是什么

NCCL = NVIDIA Collective Communications Library
它是 NVIDIA 提供的多 GPU、多节点通信库。

主要职责：

  • GPU 之间传输数据
  • 多 GPU 梯度同步
  • 支持单机和跨节点通信
  • 自动选择 PCIe、NVLink、InfiniBand/RoCE 等链路

4.4 NCCL 常见通信操作

  • AllReduce：汇总并同步所有 GPU 的数据，常用于梯度同步
  • AllGather：收集每个 GPU 的数据并分发给所有 GPU
  • ReduceScatter：归约数据并把结果分片给各 GPU
  • Broadcast：一个 GPU 向所有 GPU 分发数据
  • Send/Receive：两个 GPU 点对点传输

数据并行训练最常见的是 AllReduce。

五，区分性能瓶颈

5.1 计算瓶颈

表现：

  • GPU 计算单元利用率高
  • 矩阵乘法或卷积占用时间长
  • 多 GPU 通信占比较低

主要关注：

  • cuBLAS/cuDNN 算法
  • Tensor Core
  • 数据类型
  • 算子融合
  • Batch Size
  • 张量布局

5.2 显存带宽瓶颈

表现：

  • 频繁读写大量数据
  • 计算量相对较少
  • GPU 算力没有充分使用

主要关注：

  • 减少中间张量
  • 算子融合
  • 连续内存访问
  • 更高效的数据布局
  • 避免不必要的数据复制

5.3 通信瓶颈

表现：

  • GPU 经常等待其他 GPU
  • AllReduce 占用大量时间
  • 增加 GPU 后性能提升不明显

主要关注：

  • NCCL 通信效率
  • PCIe/NVLink/网络拓扑
  • 通信与计算重叠
  • 梯度分桶
  • 减少通信数据量
  • 合适的并行策略
