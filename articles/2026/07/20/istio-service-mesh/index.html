Istio：服务网格

Istio 是运行在 Kubernetes 上的 Service Mesh（服务网格）。它把服务之间通信需要的安全、流量控制和可观测性从业务代码中抽离出来。

一、为什么需要 Istio

假设有如下调用链：

只依靠 Kubernetes Service，通常只能解决：

  • 服务发现
  • 基础四层负载均衡
  • Pod 故障后重新选择实例

而生产系统还经常需要：

  • 服务间自动 mTLS 加密与身份认证
  • 灰度发布：5% 流量进入新版本
  • 超时、重试、熔断
  • 按 HTTP 路径、Header、用户分流
  • 限制“哪个服务可以访问哪个服务”
  • 统一收集调用成功率、延迟、流量
  • 入口和出口流量治理

没有 Istio，这些能力通常需要每种语言分别实现。Istio 将它们放到基础设施层，应用一般无须修改代码。但 Istio 不是所有系统都需要。服务数量很少、调用关系简单、没有零信任或灰度需求时，它带来的运维复杂度可能大于收益。

二、整体架构

Istio 分为：

  • 控制面：生成、下发配置，不承载业务流量。
  • 数据面：真正拦截、加密、转发业务流量。

2.1 请求路径

Sidecar 在每个工作负载内处理 L4/L7 流量。Ambient 由 ztunnel 处理 L4 流量，目标服务的 Waypoint 处理 L7 流量。

三、核心组件

3.1 Istiod

Istio 的控制面核心组件，负责：

  • 监听 Kubernetes 中的 Service、Pod、Endpoint
  • 读取 VirtualService、DestinationRule、AuthorizationPolicy 等配置
  • 通过 xDS 把配置推送给代理
  • 为工作负载签发和轮换 mTLS 证书
  • 管理 Sidecar 自动注入 Webhook

早期 Istio 的 Pilot、Citadel、Galley 等组件，现在已经合并进 Istiod。

Istiod 不转发业务流量；它短暂故障时，现有代理通常仍能按已有配置继续工作，但新配置、证书和服务变化可能无法及时同步。

3.2 Envoy Sidecar

传统 Sidecar 模式下，每个业务 Pod 内都会附加一个 Envoy：

默认情况下，应用的 TCP 入站和出站流量会被透明地导向 Envoy。UDP、ICMP 和被显式排除的端口不会经过 Envoy。Sidecar 不提供绝对的网络隔离边界。

优点：

  • 功能完整
  • 工作负载级隔离
  • L4/L7 治理能力成熟

缺点：

  • 每个 Pod 都增加 CPU、内存
  • 注入后通常需要重建 Pod
  • 代理升级和应用生命周期耦合

3.3 Istio CNI

运行在每个节点上，负责配置网络规则，将应用流量透明地导向代理。

在 Sidecar 模式中，它是可选组件，用于替代每个 Pod 内的 istio-init。业务 Pod 无须配置 NET_ADMIN、NET_RAW 权限。

在 Ambient 模式中，它负责：

  • 识别加入网格的 Pod
  • 把流量导向本节点的 ztunnel
  • 应用增删时更新流量拦截规则

Istio CNI 在 Ambient 模式中是必需组件。

3.4 ztunnel

Ambient 模式下运行在每个节点上的 DaemonSet，一个节点通常一个。

负责四层能力：

  • 服务身份
  • mTLS 加密
  • TCP 流量转发
  • L4 AuthorizationPolicy
  • TCP 指标
  • 使用 HBONE 隧道传输网格流量

ztunnel 比 Envoy 更轻量，只处理 L4 流量，不解析 HTTP。按 URL、Header 分流等 L7 功能由 Waypoint 处理。

3.5 Waypoint Proxy

Ambient 模式中可选的 Envoy L7 代理，通常按命名空间或服务部署。

负责：

  • HTTP/gRPC 路由
  • 按路径、Header 分流
  • L7 授权
  • 重试、超时
  • 灰度发布
  • HTTP 指标

3.6 Egress Gateway

管理网格访问外部系统的流量，例如：

适用于：

  • 统一出口 IP
  • 外部访问审计
  • 限制服务访问互联网
  • 对外部流量应用 TLS 和策略

它是可选组件。

Egress Gateway 不强制应用流量经过网关。强制出口和固定出口 IP 依赖 NetworkPolicy、防火墙、路由或云 NAT。

四、两种数据面模式如何选择

4.1 Ambient 模式

新项目通常可以优先评估 Ambient：

  • 不向每个 Pod 注入 Sidecar
  • 已运行 Pod 加入网格时不需要重启
  • ztunnel 提供基础零信任网络
  • 只给需要 L7 能力的服务部署 Waypoint
  • 数据面可以与应用独立升级

4.2 Sidecar 模式

适合：

  • 已经稳定运行 Sidecar Istio
  • 需要成熟、完整的工作负载级代理能力
  • 某些功能或扩展尚未适配 Ambient
  • 需要通过 EnvoyFilter 深度定制代理

Sidecar 和 Ambient 可以在同一个集群中共存并逐步迁移。同一个 Pod 不同时配置两种模式；同时匹配时，目前由 Sidecar 优先处理。

普通 Pod 加入 Ambient 时不需要重启；从 Sidecar 迁移到 Ambient 时，需要重建 Pod，以移除已经注入的 Sidecar。

4.3 功能差异

对比项 Sidecar Ambient
代理位置 每个 Pod 一个 Envoy 每个节点一个 ztunnel，按需部署 Waypoint
L4 能力 Envoy ztunnel
L7 能力 Envoy Waypoint
应用资源占用 每个 Pod 增加 CPU、内存 ztunnel 由节点共享
数据面升级 通常需要重建业务 Pod ztunnel、Waypoint 独立升级
EnvoyFilter 支持 Waypoint 不支持
VM 工作负载 支持 不支持
多集群 支持多种部署模型 仅支持多主、多网络模型

Ambient 多集群要求每个集群运行独立控制面，Waypoint 名称和服务作用域保持一致。Primary-Remote 和单网络模型不在当前支持范围内。

5.6 生产部署

  • Istiod 运行多个副本，Pod 分布在不同节点和可用区
  • Ingress Gateway 与控制面独立部署、独立扩缩容
  • 升级前运行 istioctl x precheck
  • 使用 revision 并行部署新旧控制面
  • Helm 安装和升级时固定每个 Chart 的 --version
  • Helm 配置保存在独立的 values 文件中
  • 分批迁移命名空间和网关
  • 保留旧 revision，完成验证后再移除

6.1 灰度发布

Ambient 模式执行这种 HTTP L7 路由需要 Waypoint。Kubernetes Gateway API 的 HTTPRoute 是主要路由方式，VirtualService 在 Ambient 中仍为 Alpha。

6.2 超时和重试

重试会增加下游请求量，错误配置会放大故障。非幂等请求通常不配置自动重试。

timeout 是一次请求的总时间上限，perTryTimeout 是单次尝试的时间上限。实际重试次数同时受两者限制。

6.3 熔断和异常实例驱逐

Istio 的熔断由连接池限制和异常实例驱逐组成。连接池限制并发连接和等待请求，outlierDetection 临时移除连续失败的实例。

retryBudget 按当前活动请求量限制并发重试，防止重试流量持续放大。

6.4 服务访问控制

使用 AuthorizationPolicy 可以表达：

  • 只允许 order 服务访问 payment
  • 只允许 GET 请求
  • 只允许访问 /api/pay
  • 默认拒绝其他请求

策略依据 Kubernetes ServiceAccount 对应的服务身份，而不是不稳定的 Pod IP。

命名空间默认拒绝：

存在 ALLOW 策略后，目标工作负载只接受规则明确允许的流量。Ambient 的 L4 策略由 ztunnel 执行，L7 策略通过 targetRefs 绑定到 Service 或 Waypoint。

JWT 认证由 RequestAuthentication 配置：

  • 无效 JWT 会被拒绝
  • 默认情况下，未携带 JWT 的请求仍可通过
  • 强制登录由 AuthorizationPolicy 校验 requestPrincipal
  • Ambient 模式需要 Waypoint 执行 JWT 和其他 L7 策略

6.5 强制 mTLS

加入 Ambient 后，网格内工作负载之间会通过 HBONE 自动使用 mTLS。但默认仍允许来自网格外或绕过代理的明文入站流量。

可以通过 PeerAuthentication 使用 STRICT 模式拒绝明文流量。Ambient 不支持 DISABLE 模式，因为 ztunnel 之间的 HBONE 通信始终使用 mTLS。

6.6 外部服务

ServiceEntry 将 Kubernetes 服务注册表之外的服务加入 Istio 服务注册表。注册后的外部服务可以使用路由、超时、TLS 和访问策略。

ServiceEntry 不会自动把流量导向 Egress Gateway。出口网关仍需要单独配置路由。
